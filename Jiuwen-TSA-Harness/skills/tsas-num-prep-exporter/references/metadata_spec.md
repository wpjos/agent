# tsas-num-prep-exporter 元信息格式

> 本文档定义 exporter 接受的"元信息 MD"格式。AI 负责把用户的任意输入转写为此格式后传给脚本。

---

## 1. 总体约定

- 文件为 UTF-8 编码的 Markdown
- 二级标题（`##`）作为字段锚点，**标题文字必须完全匹配**
- 字段之间顺序不限
- 不识别的小节会被忽略，不会报错

---

## 2. 字段一览

| 字段 | 是否必须 | 用途 | 缺失行为 |
|---|---|---|---|
| `## 列清单` | **必须** | 声明输入 CSV 的列名、角色、来源，**顺序决定最终输出的业务数据列顺序** | 报错中止（退出码 2） |
| `## 标注列` | 可选 | 显式列出标注列（覆盖 `## 列清单` 中"角色"列的自动识别） | 仅以 `## 列清单` 中"角色=标注"的列为标注列 |
| `## 全流程摘要` | 可选 | 在最终报告中追加 A→E 全流程的关键结果摘要 | 报告中跳过此小节 |
| `## 备注` | 可选 | 在最终报告中追加备注段落 | 报告中跳过此小节 |

---

## 3. 字段格式

### 3.1 `## 列清单`（必须）

二级 Markdown 表格，至少包含两列（列名、角色），可选第三列（来源）。

**列顺序至关重要**：表中各列的出现顺序决定最终 CSV 中业务数据的排列顺序。
最终列序固定为：`time` → 标注列（按出现顺序）→ 业务数据列（按出现顺序）。

```markdown
## 列清单

| 列名 | 角色 | 来源 |
|---|---|---|
| time | 时间 | resampler |
| label | 标注 | label.csv |
| current_amps | 业务 | current_amps.csv |
| temperature_celsius | 业务 | temperature_celsius.csv |
| vibration_rms | 业务 | vibration_rms.csv |
```

- 表头第一列必须是"列名"，分隔行（`|---|---|`）必须存在
- "角色"识别关键字：包含"标注"或英文 `label` → 视为标注列
- "来源"列仅用于最终报告展示，不影响业务逻辑

### 3.2 `## 标注列`（可选）

显式列出标注列。出现时与 `## 列清单` 中角色识别的标注列做并集。

```markdown
## 标注列

- label
```

### 3.3 `## 全流程摘要`（可选）

如果用户能提供 A→E 全流程的关键结果摘要（比如来自 profiler/merger/cleaner/resampler 的报告），可以放入此表，会原样追加到最终报告的第 1 节。

```markdown
## 全流程摘要

| 步骤 | SKILL | 关键结果 |
|---|---|---|
| A 探查 | profiler | 4 个文件，4 个数据列 |
| B 合并 | merger | 3582 行，时间范围 2025-12-01 08:00:00 ~ 2025-12-01 08:59:59 |
| C 治理 | cleaner | 显式缺失处理 + 类型/单位/编码归一 |
| D 重采样 | resampler | 目标间隔 5s，720 个网格点 |
| E 输出 | exporter | （此行可省略，会由本工具自动补齐） |
```

### 3.4 `## 备注`（可选）

自由文本，原样追加到最终报告的末尾。

```markdown
## 备注

此数据集用于电机异常检测训练；标注列由专家手工标定。
```

## 4. 完整示例

### 示例 1：最小必需

```markdown
# 输出归一化元信息

## 列清单

| 列名 | 角色 |
|---|---|
| time | 时间 |
| label | 标注 |
| current_amps | 业务 |
| temperature_celsius | 业务 |
| vibration_rms | 业务 |
```

### 示例 2：含全流程摘要 + 备注

```markdown
# 输出归一化元信息

## 列清单

| 列名 | 角色 | 来源 |
|---|---|---|
| time | 时间 | resampler |
| label | 标注 | label.csv |
| current_amps | 业务 | current_amps.csv |
| temperature_celsius | 业务 | temperature_celsius.csv |
| vibration_rms | 业务 | vibration_rms.csv |

## 全流程摘要

| 步骤 | SKILL | 关键结果 |
|---|---|---|
| A 探查 | profiler | 4 文件，4 数据列 |
| B 合并 | merger | 3582 行，1 小时 |
| C 治理 | cleaner | 类型/单位归一，异常值检测 42 项 |
| D 重采样 | resampler | 5s 网格，720 点 |

## 备注

电机异常检测数据集 v1.0
```

### 示例 3：通过 `## 标注列` 显式声明多个标注列

```markdown
# 输出归一化元信息

## 列清单

| 列名 | 角色 |
|---|---|
| time | 时间 |
| label_a | 业务 |
| label_b | 业务 |
| current_amps | 业务 |

## 标注列

- label_a
- label_b
```

---

## 5. 给 AI 的写作要点

- 列顺序：业务数据列在元信息中的排列顺序就是最终输出顺序，按用户偏好或语义重要性排列
- 标注列：明确告知哪些列是"标签"，会被自动提到 time 之后
- 全流程摘要：如果用户提供了 profiler/merger/cleaner/resampler 报告，可以提炼关键信息填入此表；只能提供其中部分时也可只填部分行
- 备注：放置数据集背景、版本号等说明性信息
- 不要写算法实现细节——元信息只描述"输入是什么"和"最终长什么样"
